# 🚀 프로젝트: 자동화 채용 통합 플랫폼 (Job Crawler & Auto-Apply System)

## 1. 프로젝트 개요
> **"채용 공고 탐색부터 AI 스펙 분석, 지원 자동화까지 개발자를 위한 채용 통합 솔루션"**

- **목적**: 매일 반복되는 채용 공고 탐색 및 서류 준비 프로세스 자동화를 통해 구직자의 생산성 극대화
- **팀 구성**: 1인 개발 (풀스택)
- **주요 기능**:
  - 채용 플랫폼 공고 정기 크롤링 및 필터링
  - 자체 로컬 LLM(Ollama/Llama3) 기반 이력서/포트폴리오 적합도 AI 자동 분석
  - 멀티 유저 인증 및 사용자별 플랫폼 외부 계정 통합 관리

---

## 2. 사용 기술 (Tech Stack)

### Backend
- **Language**: Java 21
- **Framework**: Spring Boot 3.4
- **Database**: H2 (Dev), PostgreSQL (Prod)
- **Cache**: Redis
- **Security**: Spring Security + JWT
- **Crawling**: Playwright for Java
- **AI**: Local LLM API 연동 (Ollama)

### Frontend
- **Framework**: Next.js 16 (App Router), React 19
- **Styling**: TailwindCSS 4, Shadcn UI, base-ui

## 3. 핵심 아키텍처 및 트러블슈팅 (문제 - 생각 - 시도 - 해결)
> 서비스 개발 중 마주친 기술적 문제들과, 이를 해결하기 위한 치열한 고민의 과정입니다.

### 3.1. 동적 사이트(SPA) 크롤링 한계 돌파 및 백엔드 통합
- **🚨 문제**: 사람인, 잡플래닛 등 현대 채용 포털은 React나 Vue 기반의 SPA(Single Page Application) 형태로 렌더링되거나 동적 스크립트에 의해 공고 내용이 로드됨. 기존 Jsoup 같은 정적 HTML 파서로는 필수 공고 데이터를 크롤링할 수 없음.
- **🤔 생각**: 브라우저 자동화 도구가 필수적임. 일반적으로 크롤러는 Python/Node.js로 작성하지만, 현재 메인 서버는 Java 컨텍스트임. 크롤러를 별도 파이썬 서버로 분리하면 관리 포인트와 인프라 복잡도가 2배로 늘어나 1인 개발 유지보수성에 악영향을 줌.
- **🛠 시도**: 
  1. Selenium 대비 가볍고 비동기 처리 성능이 우수한 `Playwright for Java` 라이브러리를 단일 Spring Boot 애플리케이션 내에 직접 도입하기로 결정.
  2. 추상화된 `JobScraper` 인터페이스를 선언하고 (Strategy Pattern), 각 사이트별 크롤러 빈을 DI(의존성 주입) 받아 스케줄링 시 일괄 실행되도록 모듈화 완료.
- **💡 해결**: Python 서버를 별도로 띄우지 않고도 Java 네이티브 환경에서 JS 렌더링 기반 동적 페이지 크롤링을 성공적으로 수행. **시스템 복잡도를 모놀리스 수준으로 유지하면서도 확장성 있는 크롤링 파이프라인을 구축함.**

### 3.2. Redis 캐싱을 이용한 중복 크롤링 방지 및 DB 부하 감소 (멱등성 보장)
- **🚨 문제**: 크롤러가 매 주기마다 수십~수백 개의 공고 리스트를 긁어올 때, 이전에 수집한 공고인지 매번 RDBMS(`jobPostingRepository.existsByUrl()`)에 질의하게 되면 급격한 트래픽/DB 커넥션 병목 현상 발생. 
- **🤔 생각**: "이미 수집된 공고의 URL 해시값" 정도는 휘발성 메모리에 올려두고 확인하는 것이 압도적으로 빠르겠다. RDBMS I/O를 최소화해야 크롤러 성능이 향상될 것임.
- **🛠 시도**: Redis를 활용하여 수집된 공고별로 고유 Key(`crawled:job:{hashCode}`)를 생성하고, 해당 공고의 이력 보존을 위해 `TTL(Time-To-Live) 30일`을 부여해 Redis 메모리를 적절히 캐시용으로만 유지.
- **💡 해결**: 매 크롤링 시 RDBMS 검색 대신 단위 스레드에서 `redisTemplate.hasKey()` 로 O(1) 시간복잡도만의 중복 여부 확인 수행. 결과적으로 **중복 저장 방지율 100%를 달성하며 DB 쿼리 부하(I/O 타임)를 획기적으로 낮춤.**

### 3.3. 비용 절감 모델: 사설망 Llama3 (Ollama) 로컬 AI 연동 
- **🚨 문제**: 모든 채용 공고 스펙 정리 및 자소서/포트폴리오 생성 분석 요청을 OpenAI(ChatGPT) API로 전송할 경우 API 과금 비용이 빠르게 쌓이는 문제 발생. 단순 테스트 단계임에도 비용 변수가 커짐.
- **🤔 생각**: 보안과 과금 문제를 동시에 해결하기 위해서는 온프레미스(Local)로 구동 가능한 Llama3 같은 경량화 모델을 로컬 환경 서버 (또는 자체 제공 서버)로 서빙하여 애플리케이션에서 호출하는 방식이 이상적임.
- **� 시도**: `localhost:11434/api/generate`를 통해 로컬에 구동된 오픈소스 LLM(Ollama 기반 Llama3)으로 API 엔드포인트를 연결. `AiAutomationService`를 구축하여, 채용 공고의 요구사항 텍스트와 유저의 깃허브 URL 정리본을 Prompt 단위로 주입해 적합률 매칭을 산출하는 로직 구현.
- **💡 해결**: OpenAI API 등 상용 서비스에 종속되지 않고, 독립적이고 무료인 AI 자동화 환경을 성공적으로 연동. 유저 데이터(자소서, 포트폴리오) 외부 유출을 방지할 뿐만 아니라 자유로운 프롬프트 실험을 수행할 수 있는 기반 확보.

## 4. 향후 성장을 위한 과제 방향성 (Future Improvements)
- **크롤링 스케줄링 동시성 제어**: 환경이 다중 서버로 확장될 경우 스케줄 중복 실행 방지를 위한 Redis 기반 분산 락(Redisson) 도입 필요.
- **Fetch Join 및 인프라 최적화**: API 고도화 단계에서 연관 테이블(TechStack, Requirements 등) 릴레이션이 증가함에 따른 N+1 문제 식별 시, Batch Size 조절 및 QueryDSL 최적화 적용 구상.
- **비동기 이벤트 처리**: 긴 시간을 점유하는 AI 분석 프로세스를 메인 톰캣 스레드에서 분리하기 위해 알람 이벤트를 Kafka 혹은 `@Async` 풀을 활용하는 백그라운드 태스크 시스템으로 고도화 예정.
