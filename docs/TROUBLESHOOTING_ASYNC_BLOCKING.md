# 비동기 AI 파이프라인에서 발생한 트랜잭션 전파 충돌과 순서 의존성 해결

> 이끼잡 프로젝트 — 다단계 AI 생성 파이프라인 안정화

---

## 문제 상황

자소서와 포트폴리오를 동시에 비동기 생성하는 파이프라인에서, AI 기업 분석 결과를 공유 캐시로 활용하는 구조를 도입한 뒤 **간헐적으로 전체 생성이 실패**하는 현상이 발생했다. 특히 동일 공고에 대해 처음 생성할 때만 실패하고, 두 번째부터는 성공하는 패턴이었다.

### 에러 로그
```
ERROR: cannot execute INSERT in a read-only transaction
WARN: Transaction silently rolled back because it has been marked as rollback-only
```

AI 호출 자체는 성공했지만 DB 저장 시점에서 롤백되어, 유저에게는 "생성 실패"로 표시되었다.

---

## 원인 분석

### 1단계: 트랜잭션 전파 충돌

서비스 클래스에 `@Transactional(readOnly = true)`가 클래스 레벨로 선언되어 있었고, 쓰기가 필요한 메서드는 `@Transactional`로 개별 오버라이드하는 구조였다. 기업 분석 자동 생성 로직을 추가하면서 **내부 메서드 호출**이 발생했는데, Spring AOP 프록시는 같은 빈 내부 호출에서 `@Transactional` 어노테이션을 무시한다. 결과적으로 쓰기 작업이 readOnly 트랜잭션 안에서 실행되어 INSERT가 거부되었다.

단순히 "readOnly에서 INSERT 못함"이 아니라, 문제는 **트랜잭션이 rollback-only로 마킹된 이후에도 이어지는 후속 작업(자소서/포트폴리오 저장)까지 연쇄 롤백**된다는 점이었다. AI API 호출 비용은 이미 소모되었지만 결과가 저장되지 않는, 비용만 발생하고 결과는 없는 상태가 만들어졌다.

### 2단계: 비동기 스레드 간 공유 자원 경합

자소서와 포트폴리오는 각각 별도 Thread에서 비동기 생성되는데, 두 Thread가 **동시에** 기업 분석 캐시 미스를 감지하고 각각 AI 호출을 실행했다. 동일한 기업 분석이 2번 실행되어 API 비용이 2배 발생했고, 두 Thread가 동시에 같은 레코드를 INSERT하려 해서 둘 중 하나는 반드시 실패하는 구조였다.

로그를 분석하면 Thread-16과 Thread-17이 같은 시각에 동일 기업의 웹 검색을 시작한 것이 확인되었다.

### 3단계: 순서 의존성이 있는 작업의 병렬 실행

근본 원인은 **기업 분석 → 자소서 → 포트폴리오** 사이에 순서 의존성이 있음에도, 자소서와 포트폴리오를 병렬로 실행하면서 각자 선행 작업(기업 분석)을 독립적으로 수행하려 한 설계에 있었다. 캐시가 있으면 문제없지만, 캐시 미스(최초 생성) 시 경합이 발생하는 전형적인 **Cache Stampede** 패턴이었다.

---

## 해결 과정

### 시도 1: 기업 분석 저장을 제거하고 결과만 반환
readOnly 트랜잭션 충돌을 피하기 위해 기업 분석 결과를 DB에 저장하지 않고 메모리에서만 사용하도록 변경했다. INSERT 에러는 사라졌지만, 캐시가 되지 않아 자소서와 포트폴리오에서 **매번 AI를 다시 호출**하는 문제가 발생했다. API 비용이 3배가 되어 실용성이 없었다.

### 시도 2: 별도 빈으로 분리
기업 분석 로직을 별도 @Service로 추출하여 프록시 내부 호출 문제를 해결하려 했다. 트랜잭션 문제는 해결되었지만, 여전히 두 Thread가 동시에 호출하는 경합 문제는 남아있었고, 빈 분리로 인한 의존성 복잡도가 증가했다.

### 최종 해결: 선행 작업 분리 + 순차 실행 보장

비동기 스레드 진입 직후 **기업 분석을 1회 선행 실행**하고, 그 결과를 DB에 저장한 뒤 자소서/포트폴리오가 캐시에서 조회하도록 구조를 변경했다.

```
[비동기 스레드 시작]
    ↓
1. 기업 분석 실행 + DB 저장 (1회, 외부 호출로 프록시 정상 작동)
    ↓
2. 자소서 생성 (기업 분석은 DB 캐시 조회)
    ↓
3. 포트폴리오 생성 (동일 캐시 사용)
    ↓
[결과 저장 + WebSocket 알림]
```

핵심 설계 판단:
- 기업 분석은 **인터페이스를 통한 외부 호출**로 실행하여 Spring 프록시가 @Transactional을 정상 적용
- 자소서/포트폴리오는 순차 실행하여 캐시 경합 제거
- 기업 분석 실패 시 catch에서 회사명만 사용하도록 fallback 처리하여 후속 작업은 중단 없이 진행

---

## 결과

- 기업 분석 중복 호출 제거 (2회 → 1회, API 비용 50% 절감)
- 트랜잭션 rollback-only 연쇄 실패 해소
- 최초 생성 시에도 안정적으로 동작 (Cache Stampede 방지)
- 기업 분석 실패 시에도 자소서/포트폴리오 생성은 계속 진행 (장애 격리)

---

## 배운 점

1. **비동기 환경에서 공유 자원(캐시)에 대한 경합은 설계 시점에서 제거해야 한다.** 런타임에 발생하는 Cache Stampede는 재현이 어렵고 간헐적으로 발생하여 디버깅 비용이 크다.

2. **트랜잭션 전파는 호출 경로에 따라 달라진다.** 같은 메서드라도 내부 호출과 외부 호출에서 트랜잭션 동작이 다르며, 비동기 스레드에서는 호출자의 트랜잭션을 상속받지 않는다는 점을 설계에 반영해야 한다.

3. **순서 의존성이 있는 비동기 작업은 병렬화하면 안 된다.** 전체 시간은 순차 실행이 길지만, 캐시 활용과 장애 격리 측면에서 결과 안정성이 훨씬 높다.
