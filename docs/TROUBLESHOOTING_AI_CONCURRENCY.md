# AI 동시성 제어 설계: 30명 동시 접속 환경에서의 Semaphore 고갈 방지

> 이끼잡 프로젝트 — AI 생성 파이프라인 동시성 제어

---

## 문제 상황

서비스 사용자가 30명 수준으로 늘어날 것을 대비해 부하 시나리오를 분석했다. 자소서와 포트폴리오 생성은 비동기 큐 기반으로 동작하지만, 실제 AI API 호출은 Semaphore로 동시 요청 수를 제한하고 있었다. 기존 설정은 Semaphore(5)였고, AI 호출 1건당 약 20초가 소요되었다.

### 부하 시나리오 분석

자소서/포트폴리오 생성 1건당 AI API 호출 흐름:
```
기업 분석(1회) → 자소서 1차 생성(1회) → 자소서 2차 다듬기(1회) → 포트폴리오 1차(1회) → 포트폴리오 2차(1회) = 최소 5회
```

비동기 구조 특성상 유저가 결과를 기다리는 동안 다른 공고의 생성을 추가로 요청할 수 있어, 유저 1명이 동시에 4건을 요청하는 시나리오가 현실적이었다. 30명이 각 4건씩 요청하면 Thread 120개가 생성되고, AI 호출은 최대 600회까지 발생할 수 있었다.

### 기존 구조의 한계

Semaphore(5) 기준으로 시뮬레이션한 결과:
- 5개 동시 처리, 나머지 115개 Thread 대기
- 대기 시간: 115개 × 20초 / 5 = 약 460초
- tryAcquire 타임아웃 30초 → 대부분의 요청이 타임아웃으로 실패
- 유저에게는 "AI 요청이 많아 대기 시간을 초과했습니다" 에러 반복 노출

## 원인 분석

### 1단계: Semaphore 크기와 AI 서버 처리량의 불일치

Semaphore(5)는 AI 서버가 1대일 때의 보수적 설정이었다. 하지만 AI 인프라를 확인한 결과 인스턴스를 3개까지 확장할 수 있었고, 각 인스턴스가 약 5개의 동시 요청을 처리할 수 있었다. Semaphore가 실제 처리 가능량보다 훨씬 작게 설정되어 있어 불필요한 대기가 발생하고 있었다.

### 2단계: 하드코딩된 동시성 제한

Semaphore 크기가 소스 코드에 `static final`로 하드코딩되어 있어, AI 인스턴스를 늘리거나 줄여도 서버를 재빌드하고 재배포해야만 반영할 수 있었다. 운영 중 AI 서버 상태에 따라 유연하게 대응할 수 없는 구조였다.

### 3단계: 다단계 AI 호출의 증폭 효과

기존에는 자소서 생성이 1회 AI 호출이었지만, 품질 개선을 위해 2차 다듬기 호출을 추가하면서 호출 횟수가 2배로 증가했다. 여기에 기업 분석 자동 선행 호출까지 추가되면서 공고 1건당 AI 호출이 5회로 늘어났고, 이 증폭이 Semaphore 고갈을 가속시켰다.

## 해결 과정

### 시도 1: Semaphore 단순 증가

Semaphore를 5에서 30으로 늘리는 방안을 검토했다. 하지만 AI 서버의 실제 처리 가능량을 초과하면 AI 서버 자체가 과부하로 응답 지연이 발생하고, 오히려 전체 처리 시간이 늘어나는 역효과가 예상되었다. AI 서버의 한계를 무시한 채 Semaphore만 늘리는 것은 근본 해결이 아니었다.

### 시도 2: 유저별 동시 제한

유저당 동시 1~2개로 제한하는 방안을 검토했다. 공정한 분배에는 효과적이지만, 유저별 제한만으로는 총 동시 요청 수를 AI 서버 처리량에 맞출 수 없었다. 30명 × 1개 = 30 Thread가 여전히 Semaphore를 초과할 수 있었다.

### 최종 해결: AI 인스턴스 기반 동적 Semaphore 설정

AI 인스턴스 수를 기준으로 Semaphore 크기를 산정하고, 운영 중 조정 가능한 설정값으로 외부화했다.

**산정 기준**: 인스턴스 3개 × 인스턴스당 동시 처리 5개 = Semaphore(15)

```properties
# application.properties — 운영 중 조정 가능
ai.max-concurrent-requests=15
```

```java
@Value("${ai.max-concurrent-requests:15}")
private int maxConcurrentRequests;

private Semaphore aiSemaphore;

@PostConstruct
private void initSemaphore() {
    aiSemaphore = new Semaphore(maxConcurrentRequests);
    log.info("[AI] Semaphore 초기화: 동시 요청 최대 {}개", maxConcurrentRequests);
}
```

**Semaphore(15) 기준 부하 시뮬레이션**:
- 30명 × 1건 동시 = 30 Thread, AI 호출 약 150회
- 15개 동시 처리, 15개 대기
- 대기 시간: 약 20초 (1사이클)
- tryAcquire 타임아웃 30초 이내 → 대부분 정상 처리

## 배운 점

1. **동시성 제한은 하드코딩하면 안 된다.** 인프라 구성이 바뀔 때마다 재빌드/재배포하는 것은 운영 비용이다. 설정값으로 외부화하면 운영 중 대응 속도가 달라진다.

2. **Semaphore 크기는 추측이 아니라 인프라 처리량에서 역산해야 한다.** "적당히 5개"가 아니라, AI 인스턴스 수 × 인스턴스당 처리량으로 산정해야 과부하도 대기도 최소화된다.

3. **다단계 파이프라인은 호출 횟수를 증폭시킨다.** 품질을 위해 2차 호출을 추가할 때, 동시성 제한에 미치는 영향까지 함께 계산해야 한다. 기능 추가가 인프라 부담으로 이어지는 구조를 인식하는 것이 중요하다.
