# Redis 캐시 최적화 과정 — 측정 기반 의사결정

## 배경

부하 테스트(k6, 200명 동시접속)에서 공고 API 응답 시간을 줄이기 위해 Redis 캐시 강화를 적용했다.

---

## 1단계: 현재 상태 측정 (캐시 없음)

통계 API만 Redis 5분 캐시, 나머지는 매번 DB 조회.

| API | 평균 | p95 |
|---|---|---|
| 공고 목록 | 4.9ms | 8.5ms |
| 공고 상세 | 1.7ms | 3.5ms |
| 통계 (캐시됨) | 1.0ms | 2.0ms |

DB 쿼리 자체는 EXPLAIN ANALYZE로 0.05~0.37ms. 나머지는 JPA → JSON 직렬화 + 네트워크.

---

## 2단계: 캐시 적용 대상 설계

```
공고 목록: 키워드 없는 첫 5페이지 → 1분 TTL
공고 상세: ID별 → 5분 TTL
통계: 기존 유지 → 5분 TTL
```

캐시 키 설계:
```
cache:job:list:SARAMIN:p0:s20
cache:job:detail:3706
cache:job:stats
```

무효화: 크롤링/삭제 시 목록+상세+통계 전부 삭제.

---

## 3단계: 캐시 적용 후 측정

| API | 캐시 전 p95 | 캐시 후 p95 | 변화 |
|---|---|---|---|
| 공고 목록 | 8.5ms | **10.1ms** | **+1.6ms (느려짐)** |
| 공고 상세 | 3.5ms | **2.5ms** | -1.0ms (빨라짐) |
| 통계 | 2.0ms | **1.8ms** | -0.2ms (빨라짐) |

**공고 목록이 오히려 느려졌다.**

---

## 4단계: 원인 분석

### 왜 목록 캐시가 역효과인가

```
[캐시 없이 — DB 직접 조회]
DB 쿼리: 0.1ms
JPA → Java 객체: 0.5ms
Java → JSON 응답: 1ms
합계: ~1.6ms (쿼리 자체)

[캐시 사용 — Redis 경유]
Redis GET: 0.3ms
JSON 문자열 → Redis 역직렬화: 0ms (문자열 그대로)
JSON 문자열 → Page<JobPosting> 역직렬화: 3~5ms ← 병목
합계: ~3.5ms

캐시 MISS 시:
DB 쿼리: 0.1ms + JPA 변환: 0.5ms + JSON 직렬화: 2ms + Redis SET: 0.3ms
합계: ~3ms (+ 캐시 저장 오버헤드)
```

**핵심: `Page<JobPosting>` 객체가 20건의 공고 데이터 + 페이지 메타데이터를 포함해서 직렬화/역직렬화 비용이 크다.**

DB 쿼리가 0.1ms로 이미 충분히 빠르면 Redis 경유가 오히려 느리다.

### 왜 상세 캐시는 효과가 있는가

```
[공고 상세 — 1건]
JSON 크기: ~10KB (1건)
역직렬화: 0.5ms
→ DB 쿼리(0.1ms) + JPA(0.5ms) = 0.6ms보다 약간 빠름

[공고 목록 — 20건]
JSON 크기: ~40~60KB (20건 + 페이지 메타)
역직렬화: 3~5ms
→ DB 쿼리(0.1ms) + JPA(0.5ms) = 0.6ms보다 훨씬 느림
```

**데이터 크기가 작을수록 캐시 효과가 크고, 클수록 직렬화 비용이 역전한다.**

---

## 5단계: 최종 결정

```
공고 목록: 캐시 제거 — DB가 충분히 빠름 (0.1ms)
공고 상세: 캐시 유지 — 5분 TTL, 단건이라 직렬화 가벼움
통계:     캐시 유지 — 5분 TTL, 집계 쿼리 부하 방지
```

### 클라우드 환경에서는 다를 수 있다

로컬: DB 쿼리 0.1ms (소켓 통신)
클라우드: DB 쿼리 0.1ms + **네트워크 1~5ms** = 1.1~5.1ms

클라우드에서는 DB 네트워크 지연이 추가되므로 목록 캐시도 효과가 있을 수 있다. 코드에 캐시 로직을 주석으로 남겨두고, 클라우드 배포 후 다시 측정하여 결정한다.

---

## 교훈

1. **캐시가 항상 빠른 건 아니다** — DB가 이미 충분히 빠르면 직렬화 오버헤드가 더 클 수 있다
2. **추측이 아니라 측정으로 결정** — "캐시 넣으면 빨라지겠지"가 아니라 부하 테스트 수치로 판단
3. **데이터 크기와 캐시 효율은 반비례** — 큰 객체일수록 직렬화 비용 증가
4. **환경에 따라 최적 전략이 다르다** — 로컬 vs 클라우드, SSD vs HDD, 같은 VPC vs 다른 리전
5. **캐시 무효화가 캐시 적용보다 어렵다** — 크롤링으로 데이터가 바뀔 때 어떤 캐시를 지울지가 핵심

---

## 최종 적용 상태

| 최적화 | 상태 | 효과 |
|---|---|---|
| gzip 압축 | 적용 | 네트워크 전송량 ~70% 감소 |
| HikariCP 50 | 적용 | 동시 DB 접근 여유 확보 |
| 통계 Redis 캐시 (5분) | 기존 유지 | 집계 쿼리 부하 방지, p95 1.8ms |
| 상세 Redis 캐시 (5분) | 신규 적용 | p95 3.5ms → 2.5ms |
| 목록 Redis 캐시 | **제거** | 직렬화 오버헤드 > DB 쿼리 비용 |
| 인덱스 (GIN + 복합) | 적용 | ILIKE 검색 + 사이트별 정렬 최적화 |
