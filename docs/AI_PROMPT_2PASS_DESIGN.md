# AI 자소서 다단계 프롬프트 설계 v2

> 피드백 반영: 입력 구조화 + 프롬프트 역할 분리 + 감점 제거 강화

---

## 근본 원인 분석

품질이 흔들리는 이유 (비중):

| 원인 | 비중 | 설명 |
|---|---|---|
| **입력 구조 문제** | 45% | 텍스트 덩어리로 넣으면 AI가 뭘 남기고 뭘 버릴지 판단 못함 |
| **프롬프트 문제** | 35% | 추상적 지시 위주, 삭제 기준 없음, 경험 우선순위 없음 |
| **단일 호출 구조** | 20% | 내용 선택 + 구조 설계 + 문장 정리 + 압축을 한 번에 시킴 |

## 1차 프롬프트: 초안 생성 (long_draft_prompt)

**역할**: 데이터를 빠뜨리지 않고 긴 초안 생성 + 선택 근거 제공

```text
너는 IT 기업 개발자 자기소개서 작성 코치다.

목표:
주어진 지원자 정보와 회사 정보를 바탕으로 자기소개서 초안을 작성하라.
글자수 제한은 신경 쓰지 마라. 내용을 빠뜨리지 않는 것이 우선이다.

작성 원칙:
1. 지원자의 핵심 정체성이 첫 문장에 한 줄로 박혀야 한다.
2. 추상적 미사여구 대신 실제 경험, 문제 상황, 해결 방식, 결과를 쓴다.
3. 지원자의 경험을 회사 키워드와 직접 연결한다. 연결 안 되는 경험은 쓰지 마라.
4. 허위 사실이나 과장된 표현은 쓰지 않는다.
5. 어조는 성실하고 논리적이며 과장되지 않은 톤.
6. 기업명만 바꿔도 통하는 문장은 쓰지 마라.

문단 구성:
1. 지원자 정체성 — 핵심 강점 한 줄 정의 + 뒷받침 2~3문장
2. 지원동기 — 회사 키워드와 내 경험의 접점. 찬양 금지.
3. 대표 경험 (must_include만 사용) — 각 경험마다: 문제 → 초기 시도 → 원인 분석 → 해결 → 결과 + 교훈
4. 입사 후 기여 — "성장하겠다" 금지. 구체적으로 뭘 할 수 있는지만.

경험 선택 규칙:
- must_include_experiences는 반드시 포함. 순서대로 쓴다.
- optional_experiences는 분량이 남을 때만 포함.
- must_keep_metrics의 수치는 반드시 본문에 포함.
- banned_expressions에 있는 표현은 절대 사용 금지.

출력 형식 (반드시 이 JSON 구조로 응답):
{
  "draft": "자기소개서 초안 전문 (순수 텍스트)",
  "paragraph_summary": ["1문단: ...", "2문단: ...", ...],
  "selected_experiences": [{"name": "...", "reason": "회사 키워드 OO과 연결"}],
  "discarded_experiences": [{"name": "...", "reason": "회사 키워드와 연결 약함"}],
  "weak_sentences": ["약한 문장 1", "약한 문장 2", "약한 문장 3"]
}

[지원자 핵심 정체성]
{core_identity}

[지원 회사]
{company_name} / {job_role}

[회사 키워드]
{company_keywords}

[필수 경험]
{must_include_experiences}

[선택 경험]
{optional_experiences}

[필수 수치]
{must_keep_metrics}

[금지 표현]
{banned_expressions}

[마지막 문단 목표]
{final_paragraph_goal}
```

## 2차 프롬프트: 제출형 압축 + 감점 제거 (compress_prompt)

**역할**: 톤 조정 + 글자수 + 감점 제거. 짧고 핵심만.

```text
{company_name} {job_role} 지원용 제출형으로 다시 써라.
반복 표현 줄이고, 문단별 역할 분명히 나눠라.
수치와 결과는 유지. 추상 표현은 줄여라.
톤은 과장 없이 논리적으로.
새 사실 추가 금지.
{target_length}자 이내 (공백 포함).
순수 텍스트만. 마크다운, HTML 절대 금지.

감점 제거 규칙:
- 회사명만 바꿔도 성립하는 문장 삭제.
- "성장하고 싶다", "배우겠다" 식 상투 표현 최소화.
- 구체 사례 없는 협업/소통 미화 문장 축약.
- 반복되는 추상어("구조", "안정성", "품질")는 한 번만.

[공고 자격요건 키워드]
{company_keywords}

[필수 유지 수치]
{must_keep_metrics}

[초안]
{1차 draft}
```

## 3차 프롬프트: 감점 검수 (feedback_prompt, 선택)

**역할**: 제출 전 최종 점검. 감점 요소 리포트.

```text
아래 자기소개서를 채용 담당자 관점에서 점검하라.

점검 항목:
1. 반복 추상어: "구조", "안정성", "품질" 등이 3회 이상 반복되는지
2. 상투 표현: "성장하겠다", "배우겠다", "기여하겠다" 류
3. 회사 맞춤도: 기업명만 바꿔도 통하는 문장이 있는지
4. 근거 부족: 주장은 있으나 구체적 경험/수치가 없는 문장
5. 수치 누락: 필수 수치가 본문에 빠져있는지

[필수 유지 수치]
{must_keep_metrics}

[자기소개서]
{2차 결과}

출력:
- 각 항목별 해당 문장 지적 (없으면 "없음")
- 수정 제안
```

## 파이프라인 흐름

```
[구조화된 입력 데이터]
    ↓
[1차: 초안 생성]
  - 내용 빠뜨리지 않기
  - 경험 선택 근거 JSON 포함
  - 긴 초안 (글자수 무시)
    ↓
[1차 결과에서 draft만 추출]
    ↓
[2차: 제출형 압축 + 감점 제거]
  - 톤 조정
  - 글자수 맞춤
  - 상투 표현/반복 제거
  - 6줄 프롬프트
    ↓
[최종 자소서]
    ↓
(선택) [3차: 감점 검수 리포트]
```

## 현재 구조 vs 새 구조 비교

| | 현재 | 새 구조 |
|---|---|---|
| 입력 | 텍스트 덩어리 | **구조화된 JSON** (정체성, 필수경험, 선택경험, 필수수치, 금지표현) |
| AI 호출 | 1차 생성 + 2차 다듬기 | **1차 초안(JSON) + 2차 압축 + (3차 검수)** |
| 경험 선택 | AI가 알아서 고름 | **must_include / optional로 명시** |
| 삭제 기준 | 없음 | **banned_expressions + 감점 제거 규칙** |
| 마지막 문단 | AI가 알아서 씀 | **final_paragraph_goal 명시** |
| 추적 가능성 | 결과문만 보임 | **selected/discarded 근거 JSON 제공** |

---

## 구현 우선순위

1. **입력 스키마 구조화** — Project 엔티티에 problem/initial_attempt/root_cause/action/result/lesson 필드 추가 + 프론트 입력 폼
2. **1차 프롬프트 교체** — 구조화된 데이터 기반, JSON 출력
3. **2차 프롬프트 교체** — 감점 제거 규칙 포함 6줄
4. **3차 검수** — 선택 기능으로 추가 (프론트에서 "자소서 점검" 버튼)
