# 부하 테스트 결과 (k6)

## 테스트 환경
- 머신: Apple M4 Pro, 14코어, 24GB RAM, macOS
- 서버: Spring Boot 3.4.3, Java 21
- DB: PostgreSQL 14, HikariCP 30 커넥션 (최적화 후)
- 캐시: Redis 1.3MB (공고 통계 5분 TTL)
- Rate Limiting: IP당 200회/분
- 테스트 도구: k6 (Grafana)
- 시나리오: 10→50→100→200명 점진적 증가 (80초)
- 자원 사용: Java ~273MB, PostgreSQL ~2MB, Redis ~1.3MB (합계 ~276MB)

## 테스트 대상 API
1. `GET /api/v1/jobs?page=0&size=20` — 채용 공고 목록 (페이징 + 정렬)
2. `GET /api/v1/jobs/stats` — 사이트별 공고 통계 (Redis 캐시)
3. `GET /api/v1/jobs/{id}` — 공고 상세 조회

## 테스트 1: Rate Limiting 적용 상태

| 지표 | 결과 |
|---|---|
| 총 요청 | 12,519건 (80초) |
| 성공률 | 3.19% (400건 성공) |
| 차단률 | 96.81% (Rate Limiting 정상 작동) |
| 성공 요청 평균 응답 | 3.87ms |
| 성공 요청 p95 | 10.32ms |
| 초당 처리량 | 153 req/s |

**분석:**
- IP당 분당 200회 제한이 정상 작동하여 초과 요청을 429로 차단
- 제한 내 요청은 평균 3.87ms로 빠르게 처리
- Rate Limiting이 JWT 인증 필터 앞단에 배치되어 인증 비용 없이 빠르게 차단

## 테스트 2: Rate Limiting 해제 (순수 서버 성능)

| 지표 | 결과 |
|---|---|
| 총 요청 | 12,477건 (80초) |
| 성공률 | 99.72% |
| 에러율 | 0% |
| 평균 응답 | 2.43ms |
| p95 응답 | 5.53ms |
| 최대 응답 | 28ms |
| 최대 동시 접속 | 200명 |
| 초당 처리량 | 153 req/s |
| 전송 데이터 | 521MB |

**API별 응답 시간:**

| API | 평균 | p90 | p95 |
|---|---|---|---|
| 공고 목록 (JPA 페이징) | 4.4ms | 5.8ms | 7.3ms |
| 통계 조회 (Redis 캐시) | 1.1ms | 1.7ms | 2.7ms |
| 공고 상세 (단건 조회) | 1.6ms | 2.5ms | 3.6ms |

**분석:**
- 200명 동시 접속에서도 p95 응답 5.53ms — 사용자 체감 지연 없음
- 통계 API가 가장 빠름 (1.1ms) — Redis 5분 캐시 효과
- 공고 목록이 가장 느림 (4.4ms) — JPA 페이징 + 정렬 쿼리
- 28ms 최대 응답은 GC 또는 커넥션 풀 획득 대기로 추정

## 테스트 3: 인덱스 최적화 + HikariCP 30 커넥션 적용 후

### 적용한 최적화
1. GIN 트라이그램 인덱스 추가 (title, company — ILIKE 검색 최적화)
2. 복합 인덱스 추가 (source + created_at DESC — 사이트별 정렬)
3. HikariCP maximum-pool-size: 10 → 30
4. HikariCP connection-timeout: 5000ms, idle-timeout: 300000ms

### 최적화 후 결과

| 지표 | 최적화 전 | 최적화 후 |
|---|---|---|
| 성공률 | 99.72% | **100%** |
| 에러율 | 0.28% | **0%** |
| 평균 응답 | 2.43ms | 2.49ms |
| p95 응답 | 5.53ms | 5.79ms |
| 통계 API p95 | 2.73ms | **2.09ms** (개선) |
| 공고 상세 p95 | 3.60ms | **3.55ms** (개선) |
| CPU 피크 | - | 38.8% (14코어 M4 Pro) |

### 자원 사용 (200명 동시접속 피크)
- Java CPU: 유휴 0% → 피크 38.8% (14코어 중 약 5.4코어)
- Java 메모리: ~273MB RSS
- PostgreSQL: ~2MB
- Redis: ~1.3MB
- 합계: ~276MB

## 병목 분석

### 현재 상태
- HikariCP 30 커넥션으로 200명 동시 접속 안정 처리 (성공률 100%)
- Redis 캐시가 통계 조회 부하를 흡수 (통계 API p95 2.09ms)
- Rate Limiting이 비정상 트래픽을 앞단에서 96% 차단
- GIN 인덱스로 키워드 검색(ILIKE) 최적화

### DB 쿼리 성능 (EXPLAIN ANALYZE)
| 쿼리 | 실행 시간 |
|---|---|
| 기본 목록 (페이징+정렬) | 0.12ms |
| 사이트 필터 | 0.05ms |
| 키워드 검색 (ILIKE) | 0.37ms |
| 깊은 페이지 (OFFSET 1000) | 0.95ms |

DB 쿼리 자체는 1ms 미만. k6에서 측정한 2~5ms는 JPA→JSON 직렬화 + 네트워크 오버헤드 포함.

### 추가 개선 포인트
- 공고 목록 JSON 직렬화 최적화 (DTO Projection)
- 200명 이상 동시 접속 시 HikariCP 커넥션 경합 모니터링 (Actuator metrics)
- 현재 단일 서버 구조 → 수평 확장 시 Redis 세션/캐시 공유 구조로 전환 필요

## 결론

로컬 환경 (M4 Pro, 24GB) 기준:
- **200명 동시 접속, p95 응답 5.79ms, 성공률 100%, 에러율 0%**
- Rate Limiting: 비정상 트래픽 96% 차단, 정상 요청 평균 3.87ms
- Redis 캐시: 통계 API p95 2.09ms
- HikariCP 30 커넥션: 200명 처리 시 커넥션 경합 없음
- CPU: 200명 피크 시 38.8% (여유 있음)
- 메모리: 합계 276MB (2GB 서버에서도 운영 가능)

이 수치는 포트폴리오와 면접에서 "실제로 부하 테스트를 수행하고 병목을 분석·개선한 경험"으로 활용 가능.
