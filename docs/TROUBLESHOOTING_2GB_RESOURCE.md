# 2GB 서버 운영 설계: 리소스 경합 구간 상호 배제

> 이끼잡 프로젝트 — 제한된 인프라에서의 서비스 안정성 확보

---

## 문제 상황

배포 환경을 AWS t3.micro(2코어/2GB)로 결정한 뒤, 서비스 구성 요소별 메모리 사용량을 산정하니 여유가 거의 없었다.

### 메모리 예산 분석

| 컴포넌트 | 메모리 | 비고 |
|---|---|---|
| Spring Boot (JVM) | ~700MB | 힙 512MB + 메타스페이스 |
| Next.js (standalone) | ~150MB | PM2 제거, node 직접 실행 |
| PostgreSQL | ~300MB | shared_buffers 축소 |
| Redis | ~50MB | 세션/캐시 경량 |
| Nginx | ~10MB | 리버스 프록시 |
| OS/커널 | ~200MB | |
| **합계** | **~1.4GB** | |
| **여유** | **~600MB** | Playwright 1개 실행 시 소진 |

Playwright 브라우저 인스턴스는 1개당 300~500MB를 소비한다. 크롤링(상세 페이지 병렬 수집)과 자동지원(사이트 로그인 + 폼 작성) 모두 Playwright를 사용하므로, 두 작업이 동시에 실행되면 메모리가 2GB를 초과하여 OOM(Out of Memory)이 발생할 수 있었다.

---

## 원인 분석

### 1단계: 리소스 경합 구간 식별

Playwright를 사용하는 작업을 모두 정리했다:

| 작업 | Playwright 사용 | 실행 시점 | 메모리 |
|---|---|---|---|
| **크롤링** (4개 사이트 공고 수집) | O | 스케줄러 (cron) | 300~500MB |
| **자동지원** (사이트 폼 제출) | O | 유저 요청 | 300~500MB |
| **이력서 동기화** (사이트 이력서 업데이트) | O | 유저 요청 | 300~500MB |
| **이력서 Import** (사이트 → DB) | O | 유저 요청 | 300~500MB |

이 중 2개 이상이 동시에 실행되면 Playwright 인스턴스가 2개 이상 떠서 메모리 초과.

### 2단계: 동시 실행 가능성 분석

크롤링은 스케줄러가 특정 시각에 실행하고, 자동지원/이력서 동기화는 유저가 버튼을 누를 때 실행된다. 크롤링 시간을 새벽으로 설정하면 유저 요청과 겹칠 확률이 낮지만, **보장할 수는 없었다.** 유저가 새벽에 접속해서 자동지원을 누르면 크롤링과 동시에 실행될 수 있다.

### 3단계: OOM 발생 시 연쇄 장애

Linux에서 메모리가 부족하면 OOM Killer가 프로세스를 강제 종료한다. 어떤 프로세스가 죽을지 예측할 수 없으므로, PostgreSQL이 죽으면 데이터 정합성 문제, Spring Boot가 죽으면 서비스 전체 중단이 발생한다. 단순히 "느려지는" 수준이 아니라 **서비스 전체가 불안정해지는** 위험이 있었다.

## 해결 과정

### 시도 1: Playwright concurrency를 1로 제한

크롤링의 병렬 수집 워커를 1개로 줄여 Playwright 인스턴스가 1개만 뜨도록 했다. 크롤링 단독 실행 시에는 문제가 없었지만, 자동지원이 동시에 실행되면 여전히 인스턴스 2개가 떴다. 같은 Semaphore로 제어할 수 없었다 — 크롤링과 자동지원은 별도 호출 경로였기 때문이다.

### 시도 2: JVM 힙 축소

`-Xmx512m`으로 JVM 힙을 제한했다. 메모리 여유는 늘었지만, 공고 데이터가 많아지면 GC 빈도가 늘어 응답 지연이 발생할 수 있었다. 근본 해결이 아니라 다른 곳에서 문제가 터질 여지를 만든 셈이었다.

### 최종 해결: Playwright 작업 종류 간 상호 배제

문제는 Playwright 인스턴스 수가 아니라, **다른 종류의 작업이 동시에 브라우저 리소스를 점유하는 것**이었다. 크롤링은 내부적으로 2~3개 워커를 병렬로 돌려야 수집 속도가 유지되므로 인스턴스를 1개로 줄이는 것은 현실적이지 않았다.

그래서 인스턴스 수를 제한하는 대신, **작업 종류 간 상호 배제**를 적용했다. 크롤링이 실행 중이면 자동지원/이력서 동기화는 대기하고, 반대도 마찬가지다. 같은 종류의 작업(크롤링 내부 워커 2~3개)은 병렬 허용한다.

```java
// PlaywrightManager — 작업 종류 간 상호 배제
private final ReentrantReadWriteLock crawlingLock = new ReentrantReadWriteLock();

// 크롤링 시작 시: writeLock 획득 (자동지원 차단)
public void acquireCrawlingLock() {
    crawlingLock.writeLock().lock();
}

// 자동지원 시: readLock 시도 (크롤링 중이면 대기)
public boolean tryAcquireApplyLock(long timeout, TimeUnit unit) {
    return crawlingLock.readLock().tryLock(timeout, unit);
}
```

크롤링 중 유저가 자동지원을 요청하면:
- 크롤링 완료까지 대기 (최대 60초)
- 60초 내에 못 잡으면 "크롤링이 진행 중입니다. 잠시 후 다시 시도해주세요" 반환
- 크롤링 내부 워커(2~3개)는 평소대로 병렬 동작

### 추가 조치: PM2 제거 + standalone 빌드

Next.js의 PM2 데몬을 제거하고 `standalone` 모드로 빌드하여 systemd로 직접 관리하도록 변경했다. PM2 데몬 자체의 메모리(~50MB)를 절약하고, 프로세스 재시작은 systemd가 담당한다.

## 결과

| 항목 | 변경 전 | 변경 후 |
|---|---|---|
| 서버 스펙 | 8GB 권장 | **2GB에서 안정 운영** |
| Playwright 동시 실행 | 제한 없음 (OOM 위험) | **작업 종류 간 상호 배제 (크롤링 내부 병렬은 유지)** |
| 크롤링 + 자동지원 동시 | OOM 가능 | **상호 배제로 순차 보장** |
| PM2 | 사용 (~200MB) | **제거 (systemd 대체)** |
| JVM 힙 | 기본값 (~1GB) | **512MB 제한** |
| 월 서버 비용 | ~8만원 (t3.large) | **~1.5만원 (t3.micro)** |

## 배운 점

1. **인프라 제약은 아키텍처 결정을 바꾼다.** 8GB 서버에서는 동시 실행이 자연스럽지만, 2GB에서는 상호 배제가 필수다. 같은 코드라도 인프라에 따라 운영 전략이 달라져야 한다.

2. **OOM은 느려지는 게 아니라 죽는 것이다.** 메모리 부족 시 Linux OOM Killer가 어떤 프로세스를 죽일지 예측할 수 없다. DB가 죽으면 데이터 정합성, 앱이 죽으면 서비스 중단. 메모리 예산은 "여유 있게"가 아니라 "최악의 경우에도 초과하지 않게" 설계해야 한다.

3. **비용 최적화는 기능 제거가 아니라 실행 순서 조정이다.** 크롤링도 되고 자동지원도 된다. 동시에만 안 하면 된다. 기능을 빼지 않고 실행 타이밍만 조정해서 비용을 80% 줄인 것이 핵심이다.
